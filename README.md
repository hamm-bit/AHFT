# Motivation

Recently many trading tools that surfaced has heavily relied on agentic models for the `order viewing -> pattern matching -> strategy research -> trade execution` flow. While handing all of the quant work to AI essentially diminished the high starting point required for the construction of a autonomous trading system, this is in fact, very impractical for modern high frequency trading. Both LLMs and MCP within the agentic system introduces very high and unpredictable latency, and makes hardware mapping (to FPGAs) extremely expensive, if not impossible.

This research project aims to explore ways to make "AI will do it all" trading faster, its goal is not a retail trading bot. Ideally in the late phases of the project this tool should have properties and quality that satisfy amateur or even professional edge quant trading.

# Architectural Plan

### Phase 1:

- [ ] Construct core logic for trade execution engine and order data analysis

- [ ] Add agent control plane
    - [ ] Add interface for market API for LLM
    - [ ] Build lightweight MCP server with endpoints:
        - [ ] Fetch data from analyzed order queue
        - [ ] Construct strategy according to specific IO format
        - [ ] Access and modify strategy matrix
        - [ ] Read extra realtime information online if necessary (optional)
    - [ ] Add a guardrail to the execution engine to prevent rare but sudden hallucination over market from crashing the order 

### Phase 2:

- [ ] Speed-up
    - [ ] Try mapping the core trade execution logic to FPGA (optional)
        - This step could be finished but **isn't useful if I cannot find low-latency network access** to connect to
    - [ ] Same for mapping order fetch/send logic to FPGA (optional)

- [ ] Improve MCP server latency

### Phase 3:

- [ ] Attempt small local models instead
    - [ ] Fine-tune high-performance small models with financial data
    - [ ] Retrain the model completely (advanced)
        - Maximize determinism and consistency
        - Strict strategy output formatting
        - Ditch natural language (extreme)

- [ ] Map the agent to FPGA
    - This requires accelerator-grade FPGAs (i.e. Alveo U200 or better, Versal Prime VP19 or equivalent) with HBM
    - MCP server should be partitioned between FPGA and CPU, preferably the FPGA-side MCP should have direct network IO


# The Stack

Stack could change as implementation and research proceeds

|Layer|Technology|Why?
|---|---|---|
|Control Plane|Python + MCP|Ease of integration with market APIs and LLMs.
|Data Plane|C++ or Rust|"Zero-copy memory, predictable latency, no GC pauses."
|Inter-Process|FlatBuffers|"Fast, memory-efficient serialization."
|Local Model|Mamba / Phi-3|Predictable inference time vs. large-scale LLMs.
